Revision note (v2). Design decisions D1–D4 are now resolved (§5). The biggest change: roots come from a .code-workspace file, not folder discovery (D4). Repos are arbitrary absolute paths ("cousins") with no guaranteed common parent, which retires the prototype's scan/auto-detect path and reshapes the workspace anchor and agent-cwd model. 1. Executive summary

T3 Code today assumes one project = one git repository = one working directory at every layer (orchestration, VCS, checkpointing, file browser, agent cwd). Users who work across several repos at once — a feature spanning a frontend and a backend, a ticket touching multiple microservices — want a single thread whose agent can edit across all of them and whose diff/status view shows changes from each.

A community fork (func0der/feat/multi-repo-root) built the data foundation: a project can record an array of repoRoots plus per-root repositoryIdentities. It is rebased onto current main, typechecks clean, and passes the suite. Its discovery mechanism (scan a parent folder for sibling repos) was a reasonable v1 guess but is superseded by the resolved design: the source of truth is now a user-managed .code-workspace file.

Where we are: the data spine (the repoRoots array, migration, identity resolution, contract fields) is correct and kept. Two things change: (a) how roots are sourced — parse a .code-workspace file instead of scanning a parent; and (b) every behavioral consumer must stop collapsing the array to repoRoots[0] and instead fan out across all roots (VCS, diff, checkpoints, agent context, worktrees). That fan-out is the bulk of the remaining work.

2. Problem & use cases

From #1453, the canonical request:

"I work on them through a VS Code workspace so the agents can make changes across all repositories… I can add the parent folder containing all projects, but then the git changes are not visible."

Note the user's own framing: a VS Code workspace. That is exactly the .code-workspace model we are adopting. Representative voices:

  • Microservice work — "tickets spanning multiple micro-services to be feature complete."
  • Frontend + backend in one chat — "2 different folders / different repos… work in them in the same chat."
  • Cousin layout (this design's driving case) — repos are not under a shared parent; they sit in different trees and only share a grandparent, so "add the parent folder" cannot express the set. A .code-workspace file lists them explicitly by absolute (or relative) path.

5. Design decisions resolved

Settled with the team. Each drives the phases below.

D1Agent working directory across N roots→ C: per-root context

Decision: per-root agent context. Rather than pin a single cwd, T3 hands the provider the full root set via a generated workspace manifest (a resolved .code-workspace) so the agent indexes and edits each repo directly. This is the only option that survives the cousin layout — there is no common parent to launch in.

  • cwd remains a stable anchor (the .code-workspace file's directory) for providers that require one, but the authoritative root list is the manifest.
  • In worktree mode (D3) the manifest's folder paths point at the per-root worktrees, not the originals.

Per-provider reality (verified in the protocol schemas): there is no single clean field both agents accept for "multiple roots." T3 already spawns both provider CLIs as child processes (ChildProcessSpawner), so launch flags are available. Abstract behind a "workspace manifest → provider launch args/RPCs" adapter:

ProviderSession cwdMulti-root mechanism
Codex (app-server)single cwd per conversationProtocol has real multi-root affordances: skills/extraRoots/set { roots: string[] }, fileSearch { roots: string[] }, hooks/list { cwds: string[] }. Register extra roots after session start.
Claude (ACP)session/new carries one cwd (absolute) + mcpServers — no multi-dir fieldUse the Claude Code CLI --add-dir <root> launch flag (one per extra root) when T3 spawns the agent process. Process-level, not per-session. _meta exists but agents "MUST NOT make assumptions."

Decided (Q6): native. Codex gets multi-root via RPCs (extraRoots/search roots); Claude gets it via --add-dir launch flags. Both degrade to single-cwd (anchor) if the mechanism is unavailable.

D2Per-root checkpoint refs→ normalized child table

Decision: a child table (e.g. projection_checkpoint_refs(checkpoint_id, repo_root, ref)), "unless it becomes too cumbersome." Queryable, indexable, and a clean fan-out per root. A checkpoint is the set of its per-root rows; diff = union of per-root diffs grouped by repo; restore reverts each root's ref. Single-root keeps one row — no special-casing.

D3Worktrees × multi-root→ worktree fan-out

Decision: fan out. An isolated run creates one worktree per repoRoot, grouped under a per-thread directory; the generated manifest (D1) points the agent at those worktrees so diffs/checkpoints operate on the isolated copies. Promotes worktree fan-out from "deferred" to a first-class phase (Phase 4).

  • Thread state moves from one worktreePath to a per-root map { repoRoot → worktreePath }.
  • Create/remove fan out and are transactional-ish: partial failure must clean up created worktrees.
  • Placement (decided): reuse the existing config.worktreesDir = <T3_HOME>/worktrees (already created at startup in apps/server/src/config.ts:95), grouped per thread + per repo, e.g. <worktreesDir>/<projectId>/<threadId>/<repoName>. Base stays configurable via T3_HOME (aligns with #1878).
D4How roots are defined→ .code-workspace, user-managed

Decision: no folder discovery. The set of roots is defined by a VS Code .code-workspace file, which the user manages. This is the central design change — see §6 for the full model. Rationale: real layouts have repos as cousins (different trees, only a grandparent in common), which a parent-folder scan structurally cannot express; the .code-workspace file already exists in users' VS Code flow and lists folders explicitly.

D5Root management UX→ edit the workspace file

"Manage" = view the workspace's folders, add/remove a folder, and re-resolve. Since the .code-workspace is the source of truth, T3's management UI round-trips to that file (and/or re-reads it when it changes on disk). The command plumbing (ProjectMetaUpdateCommand.repoRoots) already exists.

6. The model shift: .code-workspace as source of truth

A VS Code .code-workspace file is JSON-with-comments listing folders: 

// my-feature.code-workspace
{
  "folders": [
    { "path": "/Users/me/work/backend" },        // absolute
    { "path": "../frontend", "name": "Web UI" },  // relative to this file
    { "path": "/Users/me/oss/shared-lib" }        // a cousin, different tree
  ],
  "settings": { }
}

What changes conceptually

  • No common-parent assumption. Roots are arbitrary absolute paths. workspaceRoot stops meaning "the parent of the repos" and becomes the anchor = the directory containing the .code-workspace file (always exists; keeps the column non-null). A new optional workspaceFile field stores the file path.
  • Roots = resolved folders. Parse folders[], resolve each path (relative → against the file's dir; absolute → as-is). Among resolved folders, the git ones become repoRoots for VCS/diff/checkpoint purposes; non-git folders remain browsable and agent-visible but contribute no diff/status.
  • User-managed. Editing the file (in VS Code or via T3) changes the root set. T3 re-reads on load and on add; the management UI writes back to the file.
  • Add-project flow gains an "Open .code-workspace" source (and/or "create a workspace from these folders"), replacing the parent-scan path.
Reader requirements: tolerate comments + trailing commas (JSONC); resolve relative/absolute paths; handle missing/renamed folders gracefully (surface, don't crash); re-read on load (Q4 — no live file-watching in v1). The file is the source of truth (Q2): a small WorkspaceFile service owns parse and serialize — T3's root-management UI writes back to the file, round-tripping unknown keys like settings.
Prototype impact: the live add-project flow we verified (folder scan → repo_roots) is superseded. The persisted repo_roots shape is unchanged, so Phases 2–4 are unaffected, but the entry point and the §4 "verified live" demo must be re-done against a .code-workspace flow.

7. Goals & non-goals

Goals

  • A project is defined by a .code-workspace file; its thread spans all listed repos (cousins, arbitrary paths).
  • The agent reads/edits across all roots via a generated per-root manifest (D1).
  • Diff & live status show changes from every root, grouped by repo.
  • Checkpoints capture/restore across all roots per turn (D2 child table).
  • Isolated runs fan out worktrees, one per root (D3).
  • File browser, search, and @-mentions span the roots (#923).
  • Fixes #2441 via per-root resolution.
  • Existing single-root projects keep working unchanged.

Non-goals (v1)

  • Cross-repo atomic commit / coordinated PR (commit per repo; see §12 — unanswered).
  • Authoring full VS Code .code-workspace semantics (only folders[] is consumed; settings/extensions are preserved but ignored).
  • Sidebar grouping / Spaces (#2293).
  • Per-root provider/model configuration.

Target: a fresh design branch off main, cherry-picking the kept data spine from multi-repo-root-rebased.

Phase 1aAdopt the data spinelanded

Keep the fork's model/migration/identity work (§4). Drop or quarantine the folder-scan discovery. Acceptance: typecheck + suite green; single-root behavior identical; repo_roots backfilled.

Phase 1b.code-workspace as source of truthlanded

Replace discovery with a workspace-file model (D4).

  • New WorkspaceFile service: parse JSONC, resolve folders[] → absolute roots, classify git vs non-git, serialize back (preserve unknown keys).
  • Add workspaceFile?: string to the project; redefine workspaceRoot as the file's dir; derive repoRoots from resolved git folders.
  • Add-project source: "Open .code-workspace" (file picker) + optional "create workspace from folders."
  • Re-read on load; surface missing/renamed folders.

Acceptance: opening a .code-workspace with cousin repos creates one project whose repoRoots are the resolved git folders; non-git folders listed but not in repoRoots.

Landed. Shipped end-to-end and user-reachable:

  • workspaceFile?: string threaded through contracts → decider → projector → projection → persistence; migration 034_ProjectionProjectsWorkspaceFile; surfaced in the web store.
  • RPC filesystem.readWorkspaceFile + WorkspaceFileLive wired into the runtime.

  • "Open .code-workspace" entry point in CommandPalette.tsx → pick → resolve → project.create with workspaceFile + resolved repoRoots; surfaces missing folders.
Deferred (by design): re-read on load and the management/round-trip write UI move to Phase 5 (D5); the WorkspaceFile.withFolders/write serialize path exists but is not yet wired to a consumer.

Fan git surfaces out over repoRoots (arbitrary absolute paths). Makes "git changes visible" true. D2.

Server

  • projection_checkpoint_refs child table (D2); CheckpointStore + CheckpointReactor capture/diff/restore per root; one aggregated receipt per turn.

Contracts / web

  • Diff + status payloads gain per-root grouping { root, displayName, files[] }[]; diff panel renders repo section headers.

Acceptance: a turn editing two cousin repos shows both in the diff; checkpoint restore reverts both. Fixes #2441.

Phase 3Agent context + file surfacesto build

Per-root agent context (D1) and root-aware file UIs.

  • Workspace manifest generator: build a resolved manifest from repoRoots (or worktree paths in Phase 4) feeding a per-provider adapter — Codex: skills/extraRoots/set + search roots; Claude: --add-dir <root> launch flags (Q6 = native).
  • ProviderCommandReactor: launch with the manifest; cwd = anchor (workspace-file dir); degrade to single-cwd where a provider lacks the mechanism.
  • WorkspaceEntries / WorkspacePaths: accept multiple roots; list/search union; path validation allows any root.
  • AssetAccess: resolve workspace-file against whichever root contains it.
  • @-mentions index across roots (#923).

Acceptance: agent edits a file in a cousin repo from one thread; @-mention finds files in any root; preview opens cross-root files.

Phase 4Worktree fan-outto build

Isolated runs across all roots (D3).

  • Thread state: worktreePath → { repoRoot → worktreePath } (persistence + projection).
  • vcs.createWorktree / GitVcsDriverCore: fan out create/remove; clean up on partial failure; choose a per-thread grouping dir.
  • Manifest generator (Phase 3) targets the worktree paths when in worktree mode.
  • Diff/checkpoint resolution (Phase 2) prefers the per-root worktree.

Acceptance: starting an isolated run creates N worktrees; the agent works in them; merging/cleanup removes all N.

Phase 5Root management UXto build
  • Project settings: list folders (git/non-git), add/remove, re-resolve — round-tripping to the .code-workspace (D5).
  • Handle external edits to the file (re-read / reconcile).
  • Submodule support (#480) if it falls out of per-root resolution.

10. Testing strategy

  • WorkspaceFile parser: JSONC (comments/trailing commas); relative vs absolute resolution; cousin paths; missing/renamed folder; non-git folder classification; serialize round-trip preserving settings.
  • Unit: resolveThreadRepoRoots (0/1/N roots, worktree map present); identity dedup.
  • Projection: multi-root snapshots and meta-update round-trips (ProjectionSnapshotQuery.test already carries the new fields).
  • VCS integration: two cousin repos in different trees; assert diff + status aggregate both; checkpoint capture/restore reverts both. (CI must allow safe.bareRepository — current local git config fails these tests, unrelated to this work.)
  • Worktree fan-out: create N, assert manifest targets them, remove N, assert cleanup on injected partial failure.
  • E2E: open a .code-workspace → project with resolved roots → agent turn edits two repos → diff shows both.
  • Migration: single-root DB upgrades with repo_roots backfilled; behavior unchanged.
